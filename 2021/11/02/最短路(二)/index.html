最短路(二)

2021-11-02

这篇是拿来讨论A*的启发函数的

先搬上一段从CSDN上cp来的结论（本文源于此文）：

启发函数可以用来控制A星算法的行为.

  • 在极端情况下,如果h(n)=0,那么只有g(n)实际上是有用的,这时A星算法也就是迪杰斯特拉算法,它能保证一定可以找到一条最优路径.

  • 如果h(n)总是小于(或者等于)从结点n走到目标结点的步数,那么A星算法是一定可以找到最优路径的.h(n)越小,A星扩展的结点越多,导致A星算法越慢.

  • 如果h(n)恰好等于从结点n走到目标结点的步数,A星算法扩展的所有结点都在最优路径上,它不会扩展任何其他无关结点,此时A星算法的速度是非常快的.尽管你无法总是做到这一点,但在某些特定情况下你确实做到.知道A星算法可以在某些时候运行的很好是一件很值得高兴的事.

  • 如果h(n)所给出的信息有时大于从结点n走到目标结点的步数,那么A星算法将无法确保能够找到最优路径,但它会运行得更快.

  • 在另一种极端的情况下,如果h(n)非常接近于g(n),那么只有h(n)将起作用,此时A星算法实际上变成宽度优先搜索.

也就是说，随着H(n)/G(n)的值增大，A星的功能从准确逐渐偏向于快捷，缺点也由耗时变为不准确

所以，也许我们可以在g和h中寻找一个平衡……

不同的启发函数对到终点距离估值的精确度不同，而对耗散函数得适当调整也能使A星算法在速度与准度中做好权衡

一种动态权衡的方法是,假定最优路径通过单位方格的代价为1,修正的耗散函数如下:

g’(n) = 1 + alpha * (g(n) - 1)

如果alpha=0,那么这个修正的将永远是1。这时，单位距离的代价是被忽略的，A星算法仅是运行在简单的可达方格/不可达方格的程度。如果alpha=1,此时修正的耗散方程将还是原来的耗散方程,这时A星算法效率将带来更好的效率.你可以将alpha设定为[0,1]之间的任意值.

一种构造精确启发函数的方法是,预计算最优路径上任意两个结点之间的距离.对于大多数游戏地图而言,这是行不通的.然而,找到近似于精确启发函数的方法是存在的.

  • 考虑所有可能的粗糙结点,它不是最优路径上的结点.预计算任意两个粗糙结点的最短路径.

  • 预计算任意两个拐点(关于waypoint和其他优化的文章链接)之间的最短路径.这是前面粗糙栅格法的一种泛化.

然后,把从任意结点到拐点的估计函数h’加入启发函数h中.最终的启发函数是这样子的:

h(n) = h’(n, w1) + distance(w1, w2), h’(w2, goal)

或者,你可以这样考虑更优的,但是花费计算量更大的启发函数:评价所有临近于起始结点n的结点w1,和所有临近于目标结点的结点w2.

（↑这一段两篇文章都没怎么看太明白，姑且认为w1，w2和它们的最短路径是预处理好的，日后若理解的不对再回来改。）

栅格地图的启发函数

曼哈顿距离

标准的启发函数正是曼哈顿距离。假设从一个栅格移动到相邻的栅格上的最小代价D，那么曼哈顿距离启发函数表示如下：

h(n) = D * (abs(n.x-goal.x) + abs(n.y-goal.y))

对角线距离

如果你的地图允许对角线移动,那么你需要另外一个启发函数。在对角线距离中，直线和对角线移动一个栅格的代价都是D:

h(n) = D * max(abs(n.x-goal.x), abs(n.y-goal.y))

(在对角线距离中，答案取横竖最长的那条路，相当于先走对角线，把最短的那个方向走满了再沿着长方向走直线)

如果对角线移动一个栅格的代价并不是D,而是类似于D2=sqrt(2)*D,那么上面的启发函数或许就不太适用了。文章中给出了这样的方案：

h_diagonal(n) = min(abs(n.x-goal.x), abs(n.y-goal.y))

h_straight(n) = (abs(n.x-goal.x) + abs(n.y-goal.y))

h(n) = D2 * h_diagonal(n) + D * (h_straight(n) - 2*h_diagonal(n))

其中， h_diagonal(n) = 沿着对角线移动的步数, h_straight(n) = 曼哈顿距离。这里除了不是沿自由方向移动外，算出来的就是真实的长度了。

欧几里得距离

如果你的单元可以往任意方向移动，那么你应该使用直线距离：

h(n) = D * sqrt((n.x-goal.x)^2 + (n.y-goal.y)^2)

然而,在这种情况下,由于耗散函数g与启发函数f并不匹配。因此这时A星算法是有问题的.因为欧几里得距离比起曼哈顿距离和对角线距离都更短,你得到的将仍然是最优路径,但A星算法运行时间将更长。

平方的欧几里得距离

为了避免计算量巨大的开方的计算,方法仅仅是对欧几里得距离做平方：

h(n) = D * ((n.x-goal.x)^2 + (n.y-goal.y)^2)

8行！！！这绝对会导致g和h的比例的问题！当A星算法计算f(n)=g(n)+h(n)的时候,h的平方会导致它远大于g,这意味着你得到的会是过高估计的启发函数.对于更大的h,这会使得g(n)变得无足轻重了,此时A星算法退化成贪心.

f值相同导致的平局

为了解决这个问题,我们需要调整g或者h。相对来说,调整h会更加容易.平局（即距离相等的不同答案）的打破对于每个结点的影响必须是确定的**(即:它不应该是一个随机数)**,而且它必须使得每个结点的f值互异.因为A星算法(OPEN表)按结点的f值大小排序,让每个结点的f值互异也就是说,对于有着”相同的”f值的结点,仅有一个会被扩展.

一种打破平局的方法是,稍微的改变h的(单位).如果我们让h的单位减小,,随着A星算法向目标结点扩展,f值将会增加.不幸的是,这意味着A星算法更偏向于扩展离起始结点较近的结点,而不是扩展离目标结点较近的结点.所以,我们可以将h的单位稍微增大一点(即使是0.1%),此时,A星算法将扩展目标结点旁边的结点.

heuristic *= (1.0 + p)

因子p的选择应该使得 p<(每走一步的最小代价)/(路径的期望最大值)（否则答案就有问题了）.假设你并不希望路径的步数超过1000,你可以将p设定为1/1000.打破平局的微小改变的结果是,A星算法扩展的结点远比原先的算法要少。

还有一种打破平局的方法如下：

dx1 = current.x - goal.x
dy1 = current.y - goal.y
dx2 = start.x - goal.x
dy2 = start.y - goal.y
cross = abs(dx1dy2 - dx2dy1)
heuristic += cross*0.001

在这种方法中，越靠近起点与终点连线的路径会更有优势。
